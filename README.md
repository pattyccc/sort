# 排序演算法效率視覺化比較

## 一、 專案概述
本專案是一個基於 Python 實作的排序演算法效能同步比較工具。專案在不呼叫系統內建排序函式的前提下，從零實作了三種經典排序演算法（選擇排序、泡泡排序、快速排序），並利用多執行緒 (Multi-threading) 技術，讓三種演算法獨立在不同的執行緒中同時運作。

透過 Tkinter 繪圖介面，程式將資料量化為長條圖，即時呈現演算法在「比較、交換、遞迴」時的動態過程，並搭配毫秒級計時器，直觀地展示出不同的時間複雜度在純運算與視覺化環境下的效能差異。

---

## 二、 核心演算法運作解析

### 1. 選擇排序法
* 反覆從未排序數列中尋找最小值，並將其與未排序區最左邊的數字進行交換。
* 運作機制：
  * 外層迴圈固定當前要放置最小值的索引 i。
  * 內層迴圈 j 自 i + 1 往後掃描至陣列末端，尋找真正最小值的索引 s。
  * 掃描結束後，執行 arr[i], arr[s] = arr[s], arr[i] 進行位置交換。
  * 在視覺化中，每次交換皆會將這兩個柱子高亮為黃色，以便肉眼追蹤。

### 2. 泡泡排序法
* 從第一筆資料開始，逐一比較相鄰的兩筆元素，若前者比後者大則進行交換，目的是將最大元素逐一向後「擠」到陣列最後方。
* 運作機制：
  * 外層迴圈 i 控制每一輪未排序的邊界，由後往前依序固定。
  * 內層迴圈 j 連續比較 arr[j] 與 arr[j + 1]。
  * 由於高頻率地進行相鄰交換，在畫面上會呈現連續跳動的視覺感。

### 3. 快速排序法
* 採用各個擊破與遞迴策略。
* 雙指標靠攏機制 (quick_sort_logic)：
  * 基準點選擇：嚴格遵照簡報，將陣列的最開始元素設定為基準點 (pivot = start)。
  * 指標移動順序：
    1. 右指標 (right) 先行：往左移動，直到找到比基準點「小」的數值或與左指標相撞。
    2. 左指標 (left) 後行：往右移動，直到找到比基準點「大」的數值或與右指標相撞。
  * 內部交換：在兩指標相撞前，若各自找到符合條件的數值，則進行 arr[left], arr[right] 交換並高亮。
  * 基準點歸位：當左右指標碰在一起時，尋找終止，將基準點數值與相撞位置的數值交換，此時基準點左邊皆比它小、右邊皆比它大。
  * 遞迴切分：將資料拆成左右兩個子陣列，分別再次呼叫快速排序，直到子陣列長度為 1 結束。

## 三、 多執行緒與視覺化整合技術

### 1. Multi-threading
若採用傳統的循序呼叫（先跑完 Bubble 才能跑 Selection），畫面便無法同時呈現。本專案在 start() 方法中，為三個演算法各開闢了一個獨立的執行緒 (threading.Thread)：
* 三個演算法在作業系統中同時並行運算。
* 當點擊「Start Sorting Simulations」時，三個黑色畫布會同時啟動跳動，完美實現同步視覺化比較。

### 2. 執行緒安全更新
由於 Tkinter 介面元件非執行緒安全，子執行緒不可直接修改主介面。本專案實作了 make_callbacks() 包裝器：
* 利用 root.after(0, ...) 將圖表重繪 (draw_bars) 與時間更新 (timer_func) 的任務排回主執行緒的事件迴圈中執行。
* 這確保了高頻率更新動畫時，程式介面依然穩定不崩潰。

### 3. 毫秒計時器與動畫延遲的洞察
* 即時計時器：各演算法使用 time.time() 精確計算經過的毫秒數，當該執行緒排序完畢時，計時器會停止並將字體轉為綠色。
* 延遲現象分析：
  在開啟 time.sleep 動畫延遲時，由於 Quick Sort 屬於深層遞迴演算法，頻繁地進入遞迴與移動雙指標，導致其被分配到的 time.sleep 次數被成倍放大，在畫面上看起來會比 Selection Sort 慢。
  然而，若將程式中所有 time.sleep 拔除（設為 0），Quick Sort 將在點擊的瞬間（約 0.0002 秒）秒殺全場，這成功在實測中驗證了理論複雜度 O(n log n) 與 O(n²) 的巨大鴻溝。
